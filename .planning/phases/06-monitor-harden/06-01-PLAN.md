<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| SQLite ← DuckDB | DuckDB reads from trades.db; trusts that DB is not corrupted and contains valid opportunity records |
| DuckDB query → Output | Analytics script outputs potentially large JSON; consumer (dashboard, CLI) must validate structure |
| Credential secrets | Analytics queries do NOT access credential secrets; no API keys or env vars in analytics layer |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Tampering | DuckDB SQL query | mitigate | Use parameterized queries only (DuckDB prepared statement pattern: `conn.sql("... WHERE timestamp > ?", params=[cutoff])`); never concatenate user input into SQL |
| T-06-02 | Disclosure | Analytics output JSON | mitigate | Sanitize metric values before output (strip any PII that might be in opportunity market names); never log full opportunity records |
| T-06-03 | Denial of Service | analytics.py on large DB | mitigate | Add timeout to DuckDB connection (10s max per query); log query duration; fail gracefully if timeout exceeded |
| T-06-04 | Information Disclosure | Stale metrics displayed | mitigate | Always include "last_updated_at" timestamp in analytics output; dashboard UI displays "metrics may be 60s old" message |

</threat_model>
