Add WebSocket heartbeat monitoring to detect disconnects and feed slowness within 30 seconds. When a WS feed hasn't sent a message in 30 seconds, mark all prices from that feed as stale in the shared price cache. Integrate executor to skip revalidation against stale prices, preventing losses from stale data during network issues.

Purpose: Prevent execution against outdated prices during WS outages or network slowness
Output: Heartbeat monitoring in ws_feeds.py, executor integration, continuous loop integration
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| WS feed → price_cache | Feed receives prices; marks stale flag based on message timeout; executor trusts the flag |
| Executor → stale flag | Executor checks stale flag before using price; skips if marked stale |
| FeedManager → continuous loop | Continuous task calls mark_stale_feeds(); trusts state is updated correctly |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-13 | Denial of Service | Feed falsely marked stale, blocking all execution | mitigate | Clear stale flag when feed recovers (receives any message). Monitor logging for frequent stale/recovery cycles — indicates network instability, not bot fault |
| T-06-14 | Tampering | Price data manually marked stale to prevent execution | mitigate | _stale flag is in-memory only; no persistence; can be reset on bot restart; alerts logged when stale detected |
| T-06-15 | Information Disclosure | Stale flag reveals network/feed issues to attacker | mitigate | Feed health status is acceptable to expose (e.g., in dashboard); no secrets revealed |

</threat_model>
