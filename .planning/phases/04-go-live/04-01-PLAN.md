  <name>Task 1: Create go-live pre-flight check and P&L report scripts</name>
  <files>scripts/go_live_check.py, scripts/pnl_report.py</files>
  <read_first>config.py, dashboard.py, db.py, executor.py</read_first>
  <action>
Create two utility scripts in `scripts/`:

**scripts/go_live_check.py** — Pre-flight validation script:
- Accepts `--url` argument (Railway deployment URL, default `http://localhost:8080`)
- Checks: GET `/healthz` returns 200
- Checks: GET `/status` returns valid JSON with expected fields (scanning, uptime, etc.)
- Checks: GET `/metrics` returns Prometheus text with key metric names (arb_opportunities_total, trade_executions_total)
- Prints PASS/FAIL for each check with color output
- Exit code 0 if all pass, 1 if any fail
- Uses only stdlib (urllib.request, json) — no pip dependencies

**scripts/pnl_report.py** — P&L analysis from trades.db:
- Accepts `--db` argument (path to trades.db, default `trades.db`)
- Accepts `--hours` argument (lookback window, default 48)
- Queries `opportunities` table for net_profit grouped by type (strategy)
- Queries `trades` table for execution counts, success/failure rates
- Computes: total P&L, per-strategy P&L, win rate, false positive rate (rejected / total detected)
- Outputs a formatted text report with:
  - Total P&L (net_profit sum)
  - Per-strategy breakdown
  - Trade count and success rate
  - Whether the 3 success criteria are met (net positive, <5% FP, at least 1 profitable round-trip)
- Uses only stdlib (sqlite3, argparse) — no pip dependencies

Both scripts must be executable standalone: `python scripts/go_live_check.py` and `python scripts/pnl_report.py`.
  </action>

  </verify>

Monitor the running bot for 48 hours. Check these items at 6h, 12h, 24h, and 48h marks:

**Every 6 hours:**
1. Check Railway dashboard — service is running, no crash loops
2. Check webhook channel — no LOSS_SPIKE or sustained ZERO_OPP_PERIOD alerts
3. Glance at dashboard UI (GET `/`) — P&L chart trending, trades executing

**At 24 hours:**
4. Download trades.db from Railway (or check via dashboard API):
   ```bash
   # If DATA_DIR is a Railway volume, access via Railway CLI or dashboard
   python scripts/pnl_report.py --db trades.db --hours 24
   ```
5. Check: Is total P&L net positive? If negative, review per-strategy breakdown for the losing strategy.
6. If a specific strategy is losing money: consider setting its threshold higher via env var adjustment.

**At 48 hours:**
7. Run full report:
   ```bash
   python scripts/pnl_report.py --db trades.db --hours 48
   ```
8. Verify:
   - Net positive P&L (or break-even with explanation)
   - No runaway losses on any single strategy
   - Trade execution success rate > 90%
   - No duplicate trades (idempotency working)

**Kill switch if things go wrong:**
- Set `DRY_RUN=true` in Railway env vars (immediate, no redeploy needed for next scan cycle)
- Or stop the Railway service entirely via dashboard

**Threshold adjustments (if needed):**
- `MIN_NET_ROI` — raise if false positives are high
- `MAX_TRADE_SIZE` — lower if losses per trade are too large
- `DAILY_LOSS_LIMIT` — lower if daily loss is approaching limit
