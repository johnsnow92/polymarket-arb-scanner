  <action>
Create `scripts/validation_report.py` — a comprehensive validation script that measures the 3 project success criteria.

**Inputs:**
- `--db` path to trades.db (default: `trades.db`)
- `--days` lookback window (default: 7)

**Outputs a structured report covering:**

**Criterion 1: Net positive P&L over 7-day period**
- Query `opportunities` table: `SELECT SUM(net_profit) FROM opportunities WHERE timestamp >= ?` (7 days ago)
- Break down by strategy type (opportunity `type` field)
- Break down by day (daily P&L trend)
- PASS if total > 0, FAIL otherwise

**Criterion 2: <5% false positive rate on detected opportunities**
- "False positive" = opportunity detected but trade rejected by executor (risk check, revalidation failure, depth insufficient)
- Query: count opportunities with action='executed' vs action='skipped:*' or action='rejected:*' in trades table
- If trades table does not have rejection tracking, approximate by: opportunities detected (total rows) vs trades executed (rows in trades table with status != 'failed')
- PASS if (rejected / total_detected) < 0.05, FAIL otherwise
- Note: A high rejection rate is actually GOOD (risk management working) — the "false positive" here means the scan detected something that turned out not to be a real arb at execution time (price moved, insufficient depth)

**Criterion 3: At least one profitable autonomous round-trip trade**
- A "round-trip" = buy on one platform + sell on another (or buy YES + buy NO internally) with net_profit > 0
- Query trades table for any pair of trades on the same opportunity_id with combined profit > 0
- Or simpler: any opportunity with net_profit > 0 and at least one associated trade with status='filled'
- PASS if count >= 1, FAIL otherwise

**Summary output:**
```
=== 7-DAY VALIDATION REPORT ===
Period: YYYY-MM-DD to YYYY-MM-DD

Criterion 1: Net Positive P&L .............. PASS/FAIL
  Total P&L: $XX.XX
  Daily: [day-by-day breakdown]
  Best strategy: [name] ($XX.XX)
  Worst strategy: [name] ($XX.XX)

Criterion 2: <5% False Positive Rate ....... PASS/FAIL
  Detected: N opportunities
  Executed: N trades
  Rejected at execution: N (X.X%)

Criterion 3: Profitable Round-Trip ......... PASS/FAIL
  Profitable trades: N
  Example: [type] on [market] — profit $X.XX

=== OVERALL: PASS/FAIL ===
Milestone 1 status: ACHIEVED / NOT ACHIEVED
```

Uses only stdlib (sqlite3, argparse, datetime). Must handle empty tables gracefully (report "insufficient data" rather than crash).

   ```bash
   python scripts/pnl_report.py --db trades.db --hours 24
   ```
3. **Watch for alerts:** Webhook notifications for LOSS_SPIKE, ZERO_OPP_PERIOD, or platform errors

**At Day 12 (7 days after all layers enabled):**

4. Download trades.db from Railway persistent volume
5. Run the full validation report:
   ```bash
   python scripts/validation_report.py --db trades.db --days 7
   ```
6. Review each criterion:
   - **Criterion 1 (P&L):** If negative, identify which strategy/layer is responsible. Consider disabling that layer.
   - **Criterion 2 (FP rate):** If >5%, review `MIN_NET_ROI` threshold — it may need raising. Check if false positives cluster on specific platforms.
   - **Criterion 3 (round-trip):** If zero profitable trades, check if trades are being placed at all. May be a threshold issue.

**If all 3 criteria PASS:**
- Milestone 1 is ACHIEVED
- Consider raising `MAX_TRADE_SIZE` and `DAILY_LOSS_LIMIT` gradually

**If any criterion FAILS:**
- Document which failed and why
- Identify corrective action (threshold tuning, layer disabling, bug fix)
- Run another 7-day validation after corrections

**Kill switch reminder:**

- Stop Railway service for full shutdown
