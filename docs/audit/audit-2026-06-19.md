## Executive Summary

This audit covered the full arbgrid codebase (29 strategies, 8 platform API clients, execution/risk/MM layers, backtesting, and infrastructure) via 22 source-code chunk × 2 dimension (bugs, security) sweeps followed by adversarial verification from three independent refuter agents. A total of approximately 75 findings were surfaced: 12 verified confirmed bugs, 5 verified-partial security/logic issues, 14 explicitly refuted non-bugs, and roughly 44 unverified findings with strong code evidence.

The top risk themes are: (1) **silent loss of capital** — inverted adverse-selection logic causes the market maker to quote on toxic markets and pause on healthy ones (F4), a dead floor-gate in correlated pairs always passes bad opportunities (F9), and systematic fee understatement in resolution/stale/convergence scanners means live trades execute on opportunities that are net-negative after fees; (2) **credential and key material leakage** — the SOCKS proxy path drops TLS and transmits Kalshi RSA-PSS auth headers and signature material in plaintext (F15), the worst single security finding in this codebase; (3) **execution failures** — the Matchbook live order body is missing a required field so every live Matchbook order fails with a 400/422 (F17), and the latency monitor deadlocks any call path when `GEOGRAPHIC_LATENCY_ENABLED=true` (F25). Immediate action is warranted on at least the top 10 CRITICAL/HIGH items before any increase in position sizing or platform coverage.

## CRITICAL Findings

### [CRITICAL] market_maker.py:1317-1319 — Adverse-selection detection fully inverted [VERIFIED]

**Impact**: The market maker quotes aggressively on markets being adversely selected (toxic flow) and pauses on healthy markets. This is the opposite of intended behavior and will cause sustained losses on any toxic market.

**Detail**: `should_route_externally` marks a bid fill as adverse when `mid < price` and an ask fill as adverse when `mid > price`. Both conditions are inverted: a bid fill is adverse when the mid moves *up* after the fill (i.e., `mid > price`), meaning the market moved against the MM. With this inversion, toxic flow triggers no pause, and healthy two-sided flow triggers spurious pause. `should_pause` (line 1363-1368) further detects toxicity but never calls `trigger_pause`, so even if the detection were correct, the pause timer is never armed.

**Fix**: Flip both comparison operators in the adverse-selection check (`mid < price` → `mid > price` for bid fills, `mid > price` → `mid < price` for ask fills) and add a `self.trigger_pause()` call inside `should_pause` when the condition is met.

### [CRITICAL] ws_feeds.py:362-364 — SOCKS proxy silently drops TLS; Kalshi credentials transmitted in plaintext [VERIFIED-ESCALATED]

**Impact**: Any deployment using a SOCKS proxy leaks Kalshi API key IDs and RSA-PSS signature material over plaintext TCP. RSA-PSS signatures are per-request but the key material and API key ID are reusable; an attacker with access to the proxy stream can replay requests or, in a worst case, reconstruct session patterns.

**Detail**: The SOCKS proxy path constructs a raw TCP socket via `socks.create_connection` and passes it directly to `websockets.connect(uri, sock=raw_socket)`. The `websockets` library, when given a pre-built socket, skips the TLS upgrade even though the URI begins with `wss://`. Kalshi auth headers (`Kalshi-Access-Key`, `Kalshi-Access-Signature`) are then transmitted in plaintext through the proxy. This is worse than originally classified — the leaked RSA-PSS signature over the timestamp+nonce is usable as a replay within the API's nonce-tolerance window.

**Fix**: Replace the raw socket hand-off with `websockets.connect(uri, sock=raw_socket, ssl=ssl.create_default_context())` to ensure TLS wraps the proxied socket.

### [CRITICAL] matchbook_api.py:306-311 — Live order POST body missing `market-id`; every live Matchbook order fails [VERIFIED]

**Impact**: Every live Matchbook order submission returns 400/422 and is rejected by the API. The dry-run path is unaffected (it correctly includes `market-id`), so the bug is invisible in testing. Any capital allocated to Matchbook live execution is never deployed.

**Detail**: The `place_order` method builds the POST body from a dict that includes `runner-id`, `side`, `price`, and `size` but omits the required `market-id` field. The Matchbook REST API requires `market-id` at the order level to route the order to the correct market. The dry-run branch at line 300-305 does include it, creating a silent live/dry-run divergence.

**Fix**: Add `"market-id": market_id` to the live-path POST body dict alongside the existing fields.

### [CRITICAL] latency_monitor.py:370-374 — Deadlock: `should_route_externally` and `get_best_region` both acquire `self._lock` [VERIFIED]

**Impact**: Any call to `should_route_externally` when `GEOGRAPHIC_LATENCY_ENABLED=true` deadlocks the calling thread permanently. In continuous mode this blocks the WS execution path indefinitely, halting all latency-optimized order routing.

**Detail**: `should_route_externally` acquires `self._lock` (a non-reentrant `threading.Lock`) and then calls `get_best_region`, which also acquires `self._lock`. Python's `threading.Lock` is not reentrant — the second acquisition in the same thread blocks forever. The feature flag defaults to false, but any production deployment that enables geographic latency optimization will encounter this.

**Fix**: Convert `self._lock` to `threading.RLock` (reentrant lock) or refactor `get_best_region` into an internal `_get_best_region_locked` that assumes the lock is already held.

### [CRITICAL] correlated.py:449-457 — Layer 4 floor gate is dead code; all bad opportunities pass through [VERIFIED]

**Impact**: The 10% spread-collapse gate that should filter out correlated-pair opportunities below the profit floor is a `pass` statement. Every opportunity that fails this gate proceeds to execution regardless of expected ROI after spread collapse.

**Detail**: The gate at lines 449-457 evaluates whether the expected spread at settlement will erode the profit margin, and the intent is to drop the opportunity if so. The body of the conditional is `pass`, meaning no filtering occurs. In a correlated-pairs strategy, spread collapse at expiry is the primary source of alpha erosion — without this gate, the scanner emits opportunities with negative expected value.

**Fix**: Replace `pass` with `continue` (in a loop context) or `return None` / removal from results list to actually drop opportunities that fail the floor check.

### [CRITICAL] treasury.py:162-166 — Gas monitor exception causes fail-open; transfers execute regardless of gas cost [VERIFIED]

**Impact**: When the gas monitor raises any exception (network timeout, RPC error, parse failure), the exception is caught and `gas_ok` is set to `True`. Auto-rebalance transfers between Gemini and Polymarket then execute even when gas costs are prohibitive, eroding or eliminating transfer profit.

**Detail**: The `try/except` block around the gas check sets `gas_ok = True` in the `except` branch, which is a fail-open pattern. For a financial gating decision, the correct default on monitoring failure is fail-closed (`gas_ok = False`) or fail-with-alert. This was confirmed verified in the find phase.

**Fix**: Change the `except` branch to set `gas_ok = False` and emit a warning log; only proceed with the transfer after a manual override or monitoring recovery.

### [CRITICAL] treasury.py:148-150 — Daily cap counts failed "pending" transfers; ghost rows block legitimate transfers [UNVERIFIED]

**Impact**: Failed transfers that were recorded as "pending" but never updated to "failed" permanently consume daily transfer capacity. Over time the daily cap fills with ghost rows and all legitimate auto-rebalance transfers are blocked.

**Detail**: The daily transfer cap query counts all transfers with status `pending` for the current day. If a transfer fails after being inserted (network error, Gemini API rejection), the row is never updated to `failed` status. Each orphaned pending row consumes capacity indefinitely. This compounds with crash scenarios caught by `recovery.py` — `reconcile_orphaned_positions` does not cover treasury transfer rows.

**Fix**: Update the daily cap query to exclude rows that have been pending for more than a configurable timeout (e.g., 1 hour), or ensure the treasury transfer executor always updates status to `failed` in its error path.

### [CRITICAL] backtest.py:196-197 — Multi-leg backtest cost uses `price_a + price_b` regardless of leg count; 3-way ROI systematically wrong [VERIFIED]

**Impact**: All backtest performance metrics for triangular (3-way) arbs show inflated ROI because the third leg cost is excluded from the denominator. Any threshold calibration performed using backtest output will be systematically biased toward accepting unprofitable 3-way arbs in live trading.

**Detail**: `total_cost = price_a + price_b` is used for all multi-leg arb types regardless of whether the opportunity has 2 or 3 legs. For triangular arbs this understates total capital deployed by roughly 33%, overstating ROI proportionally. Additionally, the formula uses raw prices (0–1 range) rather than dollar amounts (prices × stake), making the denominator dimensionally inconsistent.

**Fix**: Use the opportunity's `total_cost` field directly (already computed correctly by the scanner) rather than recomputing from individual leg prices.

### [CRITICAL] backtest.py:318-322 — `net_profit_negrisk_internal` called with 2-element list; extra legs silently dropped [UNVERIFIED]

**Impact**: NegRisk internal arb backtests silently drop all legs beyond the second, producing incorrect profit figures for any NegRisk market with more than 2 outcomes. Threshold tuning using these results underestimates profitability and may suppress valid opportunities in live trading.

**Detail**: `net_profit_negrisk_internal([price_a, price_b])` is called with a hardcoded 2-element list. The actual function signature and NegRisk arb logic support N outcomes, but the backtest call only passes the first two prices regardless of how many outcomes the market has.

**Fix**: Build the price list from all legs stored in the opportunity snapshot (e.g., `opp["prices"]`) rather than hardcoding a 2-element list.

### [CRITICAL] recovery.py:262 — `token_id` set to `order_id`; hedger fetches wrong order book for partial fills [VERIFIED]

**Impact**: When the crash-recovery path instantiates a hedger for an orphaned position, it passes the platform's order ID (a string like `"PM-12345"`) as the CLOB token ID. The hedger uses this to fetch the order book for pricing the hedge leg — it fetches either a nonexistent or wrong order book, potentially placing the hedge at a wildly incorrect price or failing entirely.

**Detail**: Line 262 sets `token_id=trade.get("order_id", "")`. Polymarket CLOB token IDs are long numeric strings (e.g., `"12345678901234567890123456789012"`) distinct from order IDs. The `hedger.py` hedge function calls `fetch_clob_data(token_id)` with this value — the CLOB API returns an error or unrelated data, and the hedge either fails or executes at a bad price.

**Fix**: Change the recovery path to store and retrieve the CLOB token ID separately from the order ID; use `trade.get("token_id", "")` and ensure token IDs are persisted in the `trades` table.

## HIGH Findings

### [HIGH] market_maker.py:431-458 — Double `can_trade` check not atomic; inventory can exceed `max_inventory` [UNVERIFIED]

**Impact**: Under concurrent order fills, the market maker can accept two fills that together exceed `max_inventory`, creating unhedged directional exposure beyond the configured risk limit.

**Detail**: `can_trade` is checked twice in the fill-acceptance path with no lock held across both checks. Between the first check and the acceptance of the second fill, another thread can simultaneously pass the same check. This is a classic TOCTOU race on shared mutable state.

**Fix**: Wrap the check-and-update of inventory in a single `threading.Lock` acquire/release that covers both the `can_trade` evaluation and the inventory increment.

### [HIGH] event_monitor.py:271 — Platform's own price added to signal aggregator before divergence computed against same aggregator [VERIFIED]

**Impact**: The divergence signal for every event-monitor opportunity is attenuated because the platform price being measured for divergence is also included in the consensus it is compared against. Opportunities are systematically understated; some actionable divergences are missed entirely.

**Detail**: `add_signal(platform_name, platform_price)` is called before `compute_divergence(platform_price)`. The aggregator's consensus incorporates the platform price, pulling it toward itself. A platform priced at 0.4 against a true consensus of 0.6 will show attenuated divergence if 0.4 is included in the consensus calculation.

**Fix**: Compute divergence against the aggregator's consensus *before* calling `add_signal`, or use a separate aggregator instance that excludes the platform under evaluation.

### [HIGH] analytics.py:106 — `annual_sharpe = stddev * sqrt(252)` omits mean return; computes annualized volatility instead [VERIFIED]

**Impact**: All strategy performance reports display annualized volatility labeled as Sharpe ratio. Any strategy with high volatility but also high mean return will appear to underperform strategies with low volatility and low mean return. Risk-layer and strategy-weight decisions based on these figures are systematically misinformed.

**Detail**: The Sharpe ratio formula is `(mean_return - risk_free_rate) / stddev`, then annualized by multiplying by `sqrt(252)`. The implementation multiplies `stddev` by `sqrt(252)` and returns that value. The mean return term is entirely absent.

**Fix**: Replace with `sharpe = (mean_daily_return / stddev) * sqrt(252)` where `mean_daily_return` is computed from the same return series as `stddev`.

### [HIGH] continuous.py:1543 — `FINNHUB_API_KEY` referenced without import; `NameError` crashes news-snipe in continuous mode [UNVERIFIED]

**Impact**: Any invocation of news-snipe mode in `--continuous` crashes with an unhandled `NameError`, silently halting the continuous scan loop for that strategy. Depending on exception handling, it may crash the entire continuous process.

**Detail**: `FINNHUB_API_KEY` is referenced at line 1543 without a corresponding import from `config` or a local definition. The name is presumably meant to be `config.FINNHUB_API_KEY` or imported at the top of `continuous.py` — neither is present.

### [HIGH] resolution.py:75 — Fee computed as `0.02 * gross_profit` instead of `0.02 * price`; fee understated by ~24× [UNVERIFIED]

**Impact**: Resolution-snipe opportunities that are net-negative after platform fees appear profitable. Live trades execute at a loss. At a snipe price of 0.96 and a 2% fee, `gross_profit = 0.04` so the fee is computed as `0.02 * 0.04 = 0.0008` instead of the correct `0.02 * 0.96 = 0.0192` — a 24× understatement.

**Detail**: The fee should be applied to the notional cost (price paid), not to the gross profit margin. The profit margin for near-certain outcomes is a few cents, so the fee on profit is negligible, but the actual fee on notional is substantial and can exceed the gross profit entirely.

**Fix**: Change the fee computation to `fee = fee_rate * price` (notional cost basis) in `resolution.py:75`.

### [HIGH] stale.py:56 — Fee applied to price delta instead of notional cost; fee systematically understated [UNVERIFIED]

**Impact**: Stale-price exploitation opportunities appear more profitable than they are after fees. Live trades execute at lower actual net profit than predicted; at low delta values the fee understatement can make a losing trade appear profitable.

**Detail**: Fee is computed as `fee_rate * (stale_price - current_price)` (the delta). The correct basis is the notional amount paid (`fee_rate * stale_price`). For a 5% stale delta and 2% fee, the computed fee is `0.02 * 0.05 = 0.001` versus the correct `0.02 * stale_price ≈ 0.01` — a 10× understatement.

**Fix**: Replace delta-based fee computation with `fee_rate * price` (notional basis) consistent with how other scan modules compute fees.

### [HIGH] convergence.py:81-82 — Same fee-on-delta error as stale.py; fees understated 5-10× [UNVERIFIED]

**Impact**: Cross-platform convergence trades execute at lower net profit than predicted due to fee understatement on both legs. Systematic misclassification of marginal opportunities as profitable.

**Detail**: Same pattern as `stale.py:56` — fees computed on the convergence delta rather than on the notional buy price. For small convergence gaps (5-10%), this understates fees by 5-10×.

**Fix**: Apply fee calculation to notional cost (`buy_price`) rather than `(outlier_price - median_price)` on each leg.

### [HIGH] triangular.py:411-430 — CLOB refinement reads `_side_a`/`_price_b` keys never set; every candidate uses `other_price=0` and is wrongly dropped [UNVERIFIED]

**Impact**: The CLOB refinement stage of the triangular scanner always drops every candidate because `other_price` defaults to 0, making all 3-way arbs appear unprofitable at ask prices. The triangular scanner effectively produces zero results in Stage 2, no matter how profitable the mid-price Stage 1 candidates are.

**Detail**: The refinement code at lines 411-430 reads `opp.get("_side_a")` and `opp.get("_price_b")` to reconstruct the multi-leg structure. These keys are never set by the Stage 1 scanner — Stage 1 uses different key names. The resulting `other_price=0` causes the profit recalculation to always show the arb as unprofitable.

**Fix**: Align the key names written by Stage 1 with the key names read by Stage 2, or pass the full legs list directly to the refinement function.

### [HIGH] betfair.py:79 — `back_prices[0].get("price", 0)` defaults to 0; causes `ZeroDivisionError` in back-lay scan [UNVERIFIED]

**Impact**: A runner with a missing or malformed price field in the Betfair API response causes an unhandled `ZeroDivisionError` in the back-lay scan, crashing the entire Betfair scan for that event.

**Detail**: Probability is computed as `1 / back_prices[0].get("price", 0)`. When `price` is absent or null, the default is 0, and the division raises `ZeroDivisionError`. Betfair runners with no available back quotes (suspended, resulted, or SP-only) commonly return empty price lists.

**Fix**: Guard with `if not back_prices or back_prices[0].get("price", 0) == 0: continue` before the probability computation.

### [HIGH] smarkets.py:86 / matchbook.py:91 — `min_depth = min(0, size)` keeps depth at 0 when any runner has zero size [UNVERIFIED]

**Impact**: Market depth is always reported as 0 when any single runner has zero available size. Zero-depth opportunities pass through to execution (since the filter is `depth < MIN_LIQUIDITY`, and depth=0 may equal the minimum). Execution attempts on illiquid markets increase adverse fill risk.

**Detail**: `min_depth` should be initialized to `float("inf")` and progressively reduced with `min(min_depth, size)`. Instead it is initialized to 0 and `min(0, size)` always returns 0 for non-negative sizes. The variable never accurately reflects the minimum depth across runners.

**Fix**: Change initialization to `min_depth = float("inf")` in both `smarkets.py:86` and `matchbook.py:91`.

### [HIGH] kalshi_api.py:165 — HTTP 429 `_RateLimitError` bypasses `_circuit.record_failure()`; circuit breaker blind to rate limits [UNVERIFIED]

**Impact**: Sustained rate-limiting by Kalshi is not registered as failures in the circuit breaker. The circuit breaker never opens on rate-limit storms, allowing the scanner to continue hammering the Kalshi API until an IP ban or account suspension.

**Detail**: When a 429 response is received, the code raises `_RateLimitError` before calling `_circuit.record_failure()`. The `except _RateLimitError` path re-raises without ever recording the failure. Circuit breaker state is therefore not updated on rate-limit events.

**Fix**: Call `self._circuit.record_failure()` before raising `_RateLimitError`, or add `_RateLimitError` to the exception types that trigger `record_failure` in the circuit breaker's catch block.

### [HIGH] gemini_api.py:139-141 — `_sign_request` mutates caller's `payload_data` dict in-place; corrupts tenacity retries [UNVERIFIED]

**Impact**: On tenacity retry attempts, the payload dict has already been mutated by the first attempt (nonce appended, fields modified), causing subsequent attempts to use a payload with duplicate or stale fields. Gemini API rejects these malformed requests, exhausting retries.

**Detail**: `_sign_request` modifies `payload_data` by appending nonce and timestamp fields directly to the passed-in dict. Tenacity retries call `_sign_request` again with the same dict object, which now has the prior attempt's nonce embedded.

**Fix**: Deep-copy `payload_data` at the start of `_sign_request` and operate on the copy, returning the modified copy without mutating the caller's dict.

### [HIGH] db.py:162 — Column name interpolated into `ALTER TABLE` SQL via f-string; SQL injection vector [VERIFIED-THEORETICAL]

**Impact**: If any external data (API response field names, config values) flows into the column-name parameter, it can be used to inject arbitrary SQL into the `ALTER TABLE` statement. Currently the values appear hardcoded, but the pattern is a latent injection vulnerability.

### [MEDIUM] continuous.py:260-262 — `_calc_realized_pnl` uses fixed $1 payout regardless of trade size [UNVERIFIED]

**Impact**: Realized P&L displayed in continuous mode is incorrect for any trade not sized at exactly $1. For trades sized at the default `MAX_TRADE_SIZE` (e.g., $50 or $100), P&L is understated by 50-100×. This affects monitoring dashboards and the auto-shutdown on daily-loss-limit breach.

**Detail**: The formula `payout = 1.0 - cost` uses a hardcoded unit payout regardless of how many shares were purchased. Correct formula: `payout = shares * resolution_price - cost`, where `shares = cost / entry_price`.

**Fix**: Replace the hardcoded `1.0` payout with the actual payout computed from `trade["size"] / trade["price"] * resolution_price`.

### [MEDIUM] market_maker.py:1036 — `estimate_daily_reward` multiplies `avg_spread` by 100 before subtracting; any spread >1% yields 0 [UNVERIFIED]

**Impact**: Liquidity reward estimates are 0 for any market with a spread greater than 1%. Since prediction market spreads are typically 2-10%, virtually all reward estimates are 0. Strategy selection and sizing decisions based on reward estimates are therefore uninformed.

**Detail**: The formula `reward = base_rate * (1 - avg_spread * 100)` uses `avg_spread` in decimal form (0.03 for 3%) multiplied by 100, yielding 3. `1 - 3 = -2`, which is then floored at 0. The multiplication by 100 is erroneous; if `avg_spread` is already in decimal, the formula should be `1 - avg_spread`.

**Fix**: Remove the `* 100` multiplier: `reward = base_rate * (1 - avg_spread)`.

### [MEDIUM] resolution.py:126-133 — Naive vs aware datetime `TypeError` disables entire Kalshi resolution path [UNVERIFIED]

### [MEDIUM] negrisk.py:63-70 — CLOB fallback uses `mid + $0.01` for missing legs; only 50% real CLOB coverage [UNVERIFIED]

**Impact**: NegRisk arb opportunities with missing CLOB data for some legs use a synthetic price (`mid + $0.01`) as the ask price. This synthetic price will often not match the real ask, causing either false positives (synthetic ask too low) or missed opportunities (synthetic ask too high).
