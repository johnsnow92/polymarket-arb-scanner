## Executive Summary

**Verdict: Critical action required before further live capital deployment.**

The codebase contains 127 confirmed findings across three severity tiers. The most dangerous cluster involves financial-correctness defects that can directly misplace real money: a 100x position-sizing multiplier for near-zero-ROI events, a silent false-success return from Gemini order cancellation, broken orphan-trade hedging, and an unvalidated fund-transfer destination address. A second cluster of scan-correctness bugs systematically suppresses or generates phantom arbitrage signals (dead refinement gates, inverted fee weights, silently truncated opportunity sets). A third cluster of security findings includes SSRF exposure via unchecked API base URLs, Telegram markdown injection from external market titles, plaintext HTTP for an authenticated dashboard, and containers running as root.

## High Severity Findings

### H-01 — `_extract_net_roi` inflates position size by ~100x for small-ROI events
**File:** `position_sizer.py:354`
**Dimension:** bugs

```python
if abs(val) > 1:
    return val / 100.0
return val   # '0.47%' -> val=0.47, abs<=1, returns 0.47 (47%) not 0.0047
```

**Why it matters:** `event_monitor.py` formats `net_roi` as `f'{(net_profit/platform_price*100):.2f}%'` so a 0.5% ROI EventDivergence opportunity produces the string `'0.50%'`. `_extract_net_roi` strips the `%`, gets `val=0.50`, and because `abs(0.50) <= 1` it returns `0.50` — which the Kelly sizer treats as a 50% ROI. Position sizes are inflated by roughly 100x for any sub-1% ROI event, pushing them to the bankroll cap.

**Fix:** Remove the heuristic. Track whether the original string ended with `%` before stripping: `had_pct = raw_original.endswith('%'); val = float(raw); return val / 100.0 if had_pct else val`.

### H-02 — `cancel_order` silently returns `True` on non-cancellation API responses
**File:** `gemini_api.py:545`
**Dimension:** bugs

```python
if resp and resp.get("isCancelled"):
    return True
return resp is not None   # True for {"error": "order not found"}, {"isCancelled": false}, etc.
```

**Why it matters:** The executor and hedger treat `True` as confirmation the order is cancelled. An error response like `{"error": "order not found"}` or `{"isCancelled": false}` triggers `return resp is not None` → `True`. A live order is treated as gone; positions are left unhedged with no corrective action.

**Fix:** Replace the fallback: `logger.warning("Gemini cancel_order: isCancelled not set for %s: %r", order_id, resp); return False`.

### H-03 — Orphaned-trade hedging uses exchange order UUID as market token ID
**File:** `recovery.py:266`
**Dimension:** bugs

```python
db.log_partial_fill(
    token_id=trade.get('order_id', ''),  # exchange order UUID, not market token
    ...
)
```

**Why it matters:** `PartialFillHedger._attempt_hedge` calls `fetch_order_book(token_id)` to place the hedge sell. An exchange order UUID is completely different from a CLOB market token ID or Kalshi ticker. The API call will 404 or return the wrong market; the hedge silently fails; the position stays unhedged.

**Fix:** Look up the market token from `opportunity_id → opportunities table → _token_ids`, or add a `market_token` column to the `trades` schema.

### H-04 — Dead `clob_ok` variable makes multi-cross CLOB refinement a no-op
**File:** `scans/multi_cross.py:426`
**Dimension:** bugs

```python
clob_ok = True
# ... loop never sets clob_ok = False ...
if not clob_ok:   # DEAD — always False
    continue
```

**Why it matters:** In one-shot mode the WS price cache is empty. Every Polymarket leg silently falls back to Stage-1 mid-prices; the guard never fires. All multi-cross candidates escape to execution with unverified profit figures. The entire Stage-2 refinement is a no-op.

**Fix:** Set `clob_ok = False` in the fallback else-branch. Also call `_fetch_clob_for_market` for REST fallback when the WS cache is cold.

### H-05 — Triangular CLOB refinement is fail-open for Polymarket legs
**File:** `scans/triangular.py:407`
**Dimension:** security / bugs

```python
if not token_ids or len(token_ids) < 2:
    refined.append(o)   # passes through with stale mid-price
```

### H-11 — No Ethereum address validation before real-money fund transfers
**File:** `treasury.py:214`
**Dimension:** security

Only non-emptiness is checked before an irreversible on-chain transfer. A truncated address, wrong-chain address, or typo passes the guard. `GEMINI_DEPOSIT_ADDRESS` is read from `os.getenv` inside the function, bypassing startup-validated config.

### H-12 — Max drawdown formula is range (MAX−MIN), not peak-to-trough; test asserts wrong value and masks the bug
**File:** `scripts/analytics.py:107` (formula) and `tests/test_analytics.py:167` (test)
**Dimension:** bugs

```python
CASE WHEN trade_count = 0 THEN 0 ELSE (max_cumulative_pnl - min_cumulative_pnl) END as max_drawdown
```

For trades `[0.03, 0.01, -0.07, -0.02, 0.08]`, the true max drawdown is `0.10`; the formula gives `0.15`. The test asserts the wrong value `0.15`, masking the bug entirely.

**Fix:** `MAX(running_peak - cumulative_pnl) as max_drawdown` with `running_peak = MAX(cumulative_pnl) OVER (PARTITION BY type ORDER BY timestamp ROWS UNBOUNDED PRECEDING)`.

### H-13 — Sharpe ratio formula computes annualized volatility, not risk-adjusted return
**File:** `scripts/analytics.py:107`
**Dimension:** bugs

```python
THEN stddev * SQRT(252)   -- annualized volatility, not Sharpe
```

`avg_pnl` is computed in the same CTE but never referenced. A strategy with high returns and high variance produces the same output as one with zero returns and the same variance.

**Fix:** `(avg_pnl / NULLIF(stddev, 0)) * SQRT(252)`.
