## CRITICAL

### C-1 · `scans/bracket.py:298` — CLOB refinement inverts arb condition; drops ALL profitable bracket arbs
**Votes:** correctness=TRUE, impact=TRUE/critical, reproduce=TRUE — 3/3 confirmed

The first-pass scan correctly keeps opportunities where `spread = 1.0 - total_cost >= min_spread` (i.e., total_cost < 1.0 = profitable). The CLOB refinement step then re-prices with ask prices, computes `overround = total_cost - 1.0`, and does `if overround < 0: continue` — which drops exactly the profitable cases (overround < 0 ⟺ total_cost < 1.0). Only unprofitable overround cases survive. **BracketArb strategy (Strategy #31) is completely non-functional.**

```python
# scans/bracket.py:296-299
total_cost = sum(prices)
overround = total_cost - 1.0
if overround < 0:   # BUG: should be `if overround >= 0`
    continue
```

**Fix:** Invert condition to `if overround >= 0: continue` (keep underround, drop overround).

## HIGH

### H-1 · `latency_monitor.py:372` — Non-reentrant lock deadlock in `should_route_externally`
**Votes:** correctness=TRUE/critical, impact=TRUE/high, reproduce=TRUE/high — 3/3 confirmed

`should_route_externally()` acquires `self._lock` (line ~366) then calls `get_best_region()` (line ~372), which also tries to acquire the same `threading.Lock`. Python's `threading.Lock` is not reentrant — this deadlocks any thread on first invocation when `GEOGRAPHIC_LATENCY_ENABLED=true` and `_region_latencies` is populated. The feature defaults to false but the deadlock is certain when enabled.

**Fix:** Use `threading.RLock()` instead of `threading.Lock()` in `LatencyMonitor.__init__`, or restructure `get_best_region()` to accept a pre-acquired lock.

### H-3 · `continuous.py:257-259` — `_calc_realized_pnl` always returns wrong P&L for real trades
**Votes:** correctness=FALSE, impact=TRUE/high, reproduce=TRUE/medium — 2/3 confirmed

The trades table `size` column stores dollar amounts (executor passes `max_trade_size`, default $5.00). `total_fill_cost = sum(fill_price * size)` computes actual dollars spent, e.g., $0.48×$5 + $0.49×$5 = $4.85 for a $5 two-leg arb. Then `return 1.0 - total_fill_cost = 1.0 - 4.85 = -$3.85` — wildly wrong (actual P&L ≈ +$0.15). Every arbitrage position settled via the no-`winning_side` path (cross, betfair, smarkets, sxbet, matchbook, gemini, ibkr) records massive negative P&L in `trades.db`.

```python
# continuous.py:257-259
if winning_side is None:
    return 1.0 - total_fill_cost  # BUG: works only when total_fill_cost ≈ $0.97 (1-contract arbs)
```

**Fix:** Change to `return total_payout - total_fill_cost` where `total_payout = sum(t["size"] / fill * fill)` = sum of dollar sizes on winning legs, or: divide by trade size first to normalize to per-contract basis.

## MEDIUM

### M-1 · `db.py:557` — `get_strategy_pnl()` double-counts net_profit for multi-leg arbs
**Votes:** correctness=TRUE, impact=TRUE, reproduce=TRUE — 3/3 confirmed

The SQL query JOINs `trades` to `opportunities` and sums `o.net_profit` — since `net_profit` is on the `opportunities` table and there are N trade rows per opportunity, the `SUM` inflates net_profit by N. A 2-leg cross-platform arb gets counted twice; a 3-leg triangular arb gets tripled. The strategy P&L dashboard is systematically overstated.

```sql
-- db.py:557 (approx)
SELECT o.type, SUM(o.net_profit)  -- inflated: summed once per trade leg
FROM trades t JOIN opportunities o ON t.opportunity_id = o.id
GROUP BY o.type
```

**Fix:** Use `SELECT o.type, SUM(sub.net_profit) FROM (SELECT DISTINCT opportunity_id, net_profit FROM opportunities ...) sub`.

### M-4 · `position_sizer.py:354` — `_extract_net_roi` returns 83% for "0.83%" strings; Kelly massively over-sizes positions
**Votes:** correctness=TRUE, impact=TRUE/high, reproduce=TRUE — 3/3 confirmed

The guard `if abs(val) > 1: return val / 100.0; return val` fails for sub-1% percentage strings: "0.83%" → `val=0.83` → `abs(0.83) <= 1` → returns `0.83` (83% ROI) instead of `0.0083` (0.83%). This causes Kelly criterion to dramatically over-size positions for any opportunity with net_roi stored as a formatted percentage string below 1%.

### M-5 · `market_maker.py:1036` — `KalshiRewardTracker.estimate_daily_reward` always returns 0.0
**Votes:** correctness=TRUE, impact=TRUE/low, reproduce=TRUE — 3/3 confirmed

`avg_spread` is stored as a decimal fraction (e.g., 0.03 for 3%). The formula `(1 - avg_spread * 100)` converts it to percentage form (3.0) then subtracts from 1, giving −2.0. `max(0.0, ...)` clamps to 0.0 for any spread > 1%. Since `MM_MIN_SPREAD` defaults to 0.03 (3%), this always returns 0.0 in practice. Kalshi market making reward estimates are dead.

### M-7 · `dashboard.py:264` — non-constant-time password comparison exposes timing oracle  *(security)*
**Votes:** 2/3 confirmed (impact lens: FALSE; correctness + reproduce: TRUE), medium

`user == DASHBOARD_USER and pwd == DASHBOARD_PASS` uses Python's string equality, which short-circuits on the first differing character. An attacker with network access to the dashboard can distinguish correct vs. incorrect password prefixes via response time. Use `hmac.compare_digest()` instead.

### M-8 · `finnhub_api.py:80` — Finnhub API key sent as URL query parameter; logged by infra  *(security)*
**Votes:** 3/3 confirmed, medium

`params={"token": self.api_key}` causes the API key to appear in the request URL, which is logged by HTTP proxies, Nginx access logs, and Railway's log aggregator. Any log access exposes the key.

**Fix:** Move to `headers={"X-Finnhub-Token": self.api_key}` (supported by Finnhub API).

### L-3 · `continuous.py:849` — `_seq_counter` non-atomic despite misleading GIL comment
**Votes:** correctness=TRUE, impact=FALSE, reproduce=TRUE — 2/3 confirmed, low

### C-2 · `treasury.py:162-164` — Gas gate silently bypassed on every real money transfer
**Severity: critical**

`self.gas_monitor.should_execute(amount_usd)` passes a plain `float`, but `GasMonitor.should_execute()` expects an opportunity dict (calls `opp.get("net_profit", 0)` internally → `AttributeError` on a float). The surrounding `except Exception: gas_ok = True` catches this silently, making the gas ceiling check a no-op. **Every USDC transfer executes regardless of Polygon gas costs.**

```python
# treasury.py:162-164
try:
    gas_ok = self.gas_monitor.should_execute(amount_usd)  # float, not dict
except Exception:
    gas_ok = True  # silent fail-open
```

**Fix:** Change to `self.gas_monitor.should_execute({"net_profit": amount_usd})` or add a dedicated `GasMonitor.should_execute_transfer(usd)` method.

### C-3 · `matchbook_api.py:267-303, 342-362` — `place_order()` and `cancel_order()` bypass circuit breaker and retry logic
**Severity: critical**

Both methods call `self.session.post()` / `self.session.delete()` directly, skipping `_request()` which wires the circuit breaker and retry logic. Consequences:
- During a Matchbook outage, `place_order()` hammers the endpoint at full rate with no backoff.
- A transient network failure in `cancel_order()` leaves the order open — **an unhedged leg stays live with no retry**, converting a risk-free arb into a directional position.

**Fix:** Route both methods through `self._request()`.

### H-6 · `recovery.py:250-274` — TOCTOU between orphaned trade fetch and status update; possible double-hedge
**Severity: high**

`_convert_orphans_to_partial_fills()` acquires `db._lock`, fetches orphaned trades, releases the lock, then calls `db.log_partial_fill()` and `db.update_trade_status()` (each re-acquires `db._lock` independently). If `executor.py` or a concurrent recovery call changes a sibling trade's status in the gap between the read and the write, the same orphaned leg can be double-processed — triggering a second hedge for a leg already hedged.

### H-7 · `risk_manager.py:93-103` — Cross-platform balance check uses fixed 50/50 cost split
**Severity: high**

`half_cost = trade_cost / 2` is used to check both platform balances independently. For a cross-platform arb with highly asymmetric legs (e.g., YES@0.30 on Polymarket + YES@0.68 on Kalshi, total $0.98), the actual per-platform outlay is $0.30 and $0.68 — not $0.49 each. The check can approve a trade where one platform is $0.19 short, causing that order to fail at execution time.

### H-8 · `polymarket_api.py:309` — `get_balance()` may divide USDC by 1e6 twice; all trades rejected as underfunded
**Severity: high**

```python
# polymarket_api.py:309
return float(resp["balance"]) / 1e6
```

If `py-clob-client` already returns balance as a decimal USDC float (not raw USDC-wei), this divides by 1,000,000 again, producing a value like $0.000005 for a $5 balance. `risk_manager.py` then always sees "Insufficient Polymarket balance" and rejects every trade. Impact: all live trading silently disabled whenever `get_balance()` is called for pre-trade checks.

### H-9 · `betfair_api.py:200-409` — Four read methods bypass circuit breaker; IP-ban risk during outage
**Severity: high**

`list_markets()`, `list_runners()`, `get_current_orders()`, and `list_market_books()` call `self.session.post()` directly, not `self._request()`. The circuit breaker only records failures in `_request()`. During a Betfair outage these methods fire at full rate, never tripping the breaker — risking IP ban from Betfair's abuse protection.

### H-10 · `ws_feeds.py:116-178` — Race condition on `_pending_*_subs` lists; missed subscriptions
**Severity: high**

`update_subscriptions()` and `prune_subscriptions()` modify `_pending_poly_subs`, `_pending_kalshi_subs`, and `_pending_betfair_subs` from the scan thread, while asyncio event loop coroutines pop/clear the same lists. Python list operations are not thread-safe across the asyncio–threading boundary without a lock. Under concurrent scan + WebSocket reconnect events, subscriptions can be silently lost — price updates for active markets stop arriving, and the executor acts on stale data.
