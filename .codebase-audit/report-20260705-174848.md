**Top themes:**

- **Hard deadlocks** — two non-reentrant threading.Lock() re-entry paths (capital_optimizer.py, latency_monitor.py) hang calling threads in live mode.
- **Fee calculation is inverted** — `_select_fees()` swaps the probability weights, mis-pricing every cross-platform and exchange strategy. Financially the most dangerous confirmed bug.
- **CLOB refinement is broken in three strategies** — triangular arb, cross-all, and multi-cross stage-2 refiners fail to filter, producing false arb signals that reach the executor.
- **Security perimeter has holes** — five SSRF-vector env vars bypass `url_guard`, two kill-switch endpoints lack CSRF protection, and the Gemini withdrawal method accepts unvalidated addresses for irreversible on-chain transfers.
- **Test suite has extensive vacuous assertions** — roughly 25 test methods assert disjunctions or conditional blocks that are always satisfied, providing no regression protection for the code paths they name.

## Confirmed Findings

### CRITICAL

#### C-01 — Deadlock: `TaxOptimizer.get_harvest_candidates()` re-enters non-reentrant lock via `is_long_term()`
**File:** `capital_optimizer.py` — lines 302–314 | **Votes:** 3/3

```python
# get_harvest_candidates (line 302):
with self._lock:           # threading.Lock() — non-reentrant
    rate = (TAX_LONG_TERM_RATE
            if self.is_long_term(position_id)   # line 312 — calls is_long_term
            else TAX_SHORT_TERM_RATE)

# is_long_term (line 255):
def is_long_term(self, position_id):
    with self._lock:       # same threading.Lock() → DEADLOCK
```

**Why it matters:** Any call to `get_harvest_candidates()` hangs the calling thread permanently. The capital optimizer becomes unresponsive, stalling continuous-mode operation.

**Fix:** Change `self._lock = threading.Lock()` to `self._lock = threading.RLock()` in `TaxOptimizer.__init__`. This finding was submitted twice (lines 302 and 312); both submissions describe the same root cause.

### HIGH

#### H-01 — Fee probability weights are swapped in `_select_fees`, inverting EV estimates for all exchange strategies
**File:** `fees.py` — line 34 | **Votes:** 3/3

```python
# WRONG:
return prob_a * case_b_fees + (1.0 - prob_a) * case_a_fees
# CORRECT:
return prob_a * case_a_fees + (1.0 - prob_a) * case_b_fees
```
Every cross-platform strategy touching Betfair, Smarkets, or Matchbook is mis-priced. When `price_a < 0.5`, fees are underestimated and the scanner declares profit on losing trades. This is the highest financial-impact correctness bug.

#### H-19 — `_seq_counter` race condition in WebSocket threads — identical sequence numbers cause `TypeError` in `asyncio.PriorityQueue`
**File:** `continuous.py` — line 858 | **Votes:** 3/3

`seq = _seq_counter; _seq_counter += 1` is two bytecodes. Two WS threads can read the same value. When both push `(priority, seq, opp)`, heapq tries to compare the third element (a dict), raising `TypeError`. The opportunity is silently dropped.

**Fix:** `_seq_gen = itertools.count(); seq = next(_seq_gen)`.

#### Security (Medium)

**M-29** `config.py:9` — `load_dotenv("~/.claude/.env")` silently injects Claude Code's internal credentials into the trading process. Combined with line 740, the Claude API key becomes a publicly accessible module attribute. Fix: remove the `~/.claude/.env` dotenv load.

**M-35** `gemini_api.py:463` — No `DRY_RUN` guard in `place_order` or `cancel_order`. Any caller bypassing the executor's guard places a live order. Fix: mirror the `matchbook_api.py` pattern.

**M-36** `matchbook_api.py:44` / `gemini_api.py:80` — `MATCHBOOK_PROXY_URL` and `GEMINI_PROXY_URL` are not validated by `assert_public_url`. An attacker can redirect authenticated API traffic through the cloud metadata endpoint. Fix: validate both proxy URLs with `assert_public_url` before assigning to `session.proxies`.

**M-37** `scans/sxbet.py:55` (and `betfair.py:79`, `smarkets.py:51`, `resolution.py:170`) — IEEE NaN satisfies no numeric comparison (`nan <= 0` → False, `nan >= 1` → False), silently bypassing all price range guards. **M-40** `dashboard.py:402` — No upper bound on `Content-Length` for POST body read. An authenticated caller can claim a multi-gigabyte body, triggering an OOM allocation. Fix: `content_len = min(int(self.headers.get("Content-Length", 0)), 1_048_576)`.
