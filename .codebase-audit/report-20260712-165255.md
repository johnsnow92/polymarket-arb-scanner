# Codebase Audit Report — arbgrid / polymarket-arb-scanner
**Date:** 2026-07-12
**Scope:** 231 Python files — full scanner, execution, risk, market-making, backtesting, and infrastructure layers
**Audit tool:** codebase-audit skill (3-lens adversarial vote)

## Executive Summary

**Verdict:** The codebase has material correctness defects that affect live trading outcomes, two deadlocks that can hang the process in production, and a cluster of test-suite findings that give false confidence — tests that pass vacuously regardless of the behaviour under test. Security issues are mostly SSRF hygiene mismatches (the guard pattern exists but three endpoints skip it) and a pair of log-injection vectors. No critical authentication bypass or data-exfiltration path exists in the core trading loop.

### Five highest-priority findings

1. **TaxOptimizer deadlock** (`capital_optimizer.py:302`) — production process hangs whenever `TAX_AWARE_ENABLED=True` and a harvest candidate is evaluated.
2. **_select_fees weights swapped** (`fees.py:34`) — every cross-platform EV fee calculation is wrong; profitable opportunities appear unprofitable and are discarded.
3. **_refine_cross_all_with_clob dead list** (`scans/cross.py:621`) — the CLOB refinement step never removes sub-threshold opportunities; phantom arbs reach the executor.
4. **Triangular CLOB refinement uses other_price=0** (`scans/triangular.py:414`) — all triangular net_profit figures after CLOB refinement are fabricated (~0.5 per market); the scanner emits fraudulent triangular opportunities.
5. **DuckDB / SQLite type mismatch** (`scripts/analytics.py:68`) — analytics always returns empty results; all strategy metrics, Sharpe, and drawdown calculations are unreachable.

## Confirmed Findings

### CRITICAL

#### C-1 - TaxOptimizer.get_harvest_candidates deadlocks on non-reentrant lock
**File:** `capital_optimizer.py:302`
**Dimension:** bugs

**Evidence:**
```
self._lock = threading.Lock()   # line 208 — non-reentrant

# get_harvest_candidates (line 302) holds the lock:
with self._lock:
    for position_id, current_value in positions.items():
        if pnl_pct < -threshold:
            rate = (
                TAX_LONG_TERM_RATE
                if self.is_long_term(position_id)   # line 314 — acquires same lock
                else TAX_SHORT_TERM_RATE
            )

# is_long_term (line 256):
def is_long_term(self, position_id: str) -> bool:
    with self._lock:   # DEADLOCK — lock already held
```

**Why it matters:** `threading.Lock` cannot be acquired twice by the same thread. Whenever `TAX_AWARE_ENABLED=True` and at least one position has a loss exceeding the harvest threshold, `get_harvest_candidates` deadlocks permanently. The calling thread hangs; in continuous mode this freezes the capital-optimisation loop indefinitely.

**Fix:** Change `self._lock = threading.Lock()` to `self._lock = threading.RLock()`, or extract the lock-free inner logic into a private `_is_long_term_unlocked(position_id)` helper called from within the existing lock region.

### HIGH

#### H-1 - _select_fees probability weights swapped — all cross-platform EV fee calculations wrong
**File:** `fees.py:34` | **Dimension:** bugs

The correct EV-weighted fee is `prob_a * case_a_fees + (1-prob_a) * case_b_fees`. The code multiplies `case_b_fees` by `prob_a`, systematically overstating expected fees when `price_a` is high. Every caller (`net_profit_cross_betfair`, `net_profit_cross_smarkets`, `net_profit_cross_generic`, `net_profit_triangular`) uses this function under `FEE_MODEL=expected_value`, causing the scanner to discard genuinely profitable opportunities.
**Fix:** `return prob_a * case_a_fees + (1.0 - prob_a) * case_b_fees`

---

#### H-2 - _extract_net_roi interprets sub-1% percentage strings as decimal fractions (100x position over-sizing)
**File:** `position_sizer.py:355` | **Dimension:** bugs

`event_monitor.py` stores `net_roi` as `f"{(net_profit / platform_price * 100):.2f}%"`. A 0.5% ROI produces `"0.50%"`. After stripping `%`, `val=0.50`; since `abs(0.50) <= 1`, it is returned unchanged, telling the Kelly sizer the ROI is 50%. Position sizes are then 100x larger than intended.
**Fix:** Track whether the original string had a `%` suffix before stripping and always divide by 100 in that case.

#### H-3 - _refine_cross_all_with_clob never removes failing opportunities
**File:** `scans/cross.py:621` | **Dimension:** bugs

`refined_out` is initialised but never populated and never returned. When a candidate fails the CLOB profit check, `o` is simply not modified but remains in `opportunities` with its original inflated mid-price `net_profit`. Sub-threshold opportunities pass `filter_dust` and reach the executor. The analogous `_refine_cross_with_clob` correctly builds and returns a kept list.
**Fix:** Collect IDs to remove and splice the list in-place after the loop, mirroring `_refine_cross_with_clob`.

#### H-4 - CLOB refinement for triangular arbs always passes other_price=0
**File:** `scans/triangular.py:414` | **Dimension:** bugs

`scan_triangular` never stores `_price_a`, `_price_b`, `_side_a`, or `_side_b` in the opp dict. `_refine_triangular_with_clob` reads them with `.get(..., 0)`, always getting 0 for the non-PM leg. `net_profit_triangular` receives 0 for the second price, computing profit ≈ `1 − pm_ask − pm_fees` (~0.5 for near-50/50 markets). Nearly every triangular candidate passes with fabricated numbers.
**Fix:** Add `"_price_a": best_yes`, `"_price_b": best_no`, `"_side_a": "yes"`, `"_side_b": "no"` to the opp dict in `scan_triangular` (~line 322).

#### H-5 - CLOB fetch tasks map market keys to wrong market dicts when long leg is market_b
**File:** `scans/correlated.py:367` | **Dimension:** bugs

When `price_b < price_a`, `_long_market` is `market_b` but `_refine_correlated_with_depth` always zips `_market_key_a -> _long_market` and `_market_key_b -> _short_market`. So `clob_results[market_key_a]` contains `market_b`'s order book, inverting long/short CLOB data for roughly half of all correlated pairs.
**Fix:** Use `(opp.get("_long_leg"), opp.get("_long_market")), (opp.get("_short_leg"), opp.get("_short_market"))` to respect orientation.

#### H-6 - outcomePrices elements are strings; no float conversion before arithmetic in _calculate_optimal_quotes
**File:** `scans/rewards.py:231` | **Dimension:** bugs

Polymarket's Markets REST API returns `outcomePrices` as strings (e.g. `["0.75", "0.25"]`). `prices[0]` is the string `"0.75"`. `_calculate_optimal_quotes` performs `bid = mid_price - half_spread + skew`, raising `TypeError`. This crashes the reward scan for every market when the WebSocket price cache is cold (every scan cycle start).
**Fix:** `yes_price = float(prices[0])` (wrapped in `try/except ValueError`).

#### H-7 - LogicalArb stores only the buy-leg token IDs; sell-leg (if_market) has no token IDs
**File:** `scans/logical_arb.py:84` | **Dimension:** bugs

A LogicalArb requires two legs: buy the underpriced `then` market and sell the overpriced `if` market. Only the `then`-market's token IDs are stored under `_token_ids`. Without the sell-leg token IDs the executor has no execution path for the hedge, turning a risk-free arb into a naked directional buy.
**Fix:** Extract and store `"_token_ids_if": _extract_token_ids(if_market)` and update `_build_legs` in `executor.py` to consume both.

#### H-8 - DuckDB cannot read a SQLite trades.db — analytics always returns empty
**File:** `scripts/analytics.py:68` | **Dimension:** bugs

`trades.db` is a SQLite file (written by `db.py` via `sqlite3`). DuckDB's native format is binary-incompatible. The `duckdb.connect(db_path)` call raises `IOException`, which is swallowed by `except Exception` at line 135, logging an error and returning `[]`. Every strategy metric, Sharpe, and drawdown figure is permanently unreachable.
**Fix:** Use `sqlite3` (already in `db.py`), or load DuckDB's SQLite scanner extension: `conn.execute("INSTALL sqlite; LOAD sqlite; ATTACH '{}' AS trades (TYPE SQLITE);".format(db_path))`.

#### H-9 - Correlated opportunity _long_leg/_short_leg format mismatch between scanner output and executor test fixture
**File:** `tests/integration/test_executor_strategies.py:251` | **Dimension:** bugs

The test fixture passes `_long_leg` as a dict with embedded `_token_ids`. `scan_correlated` output (per `test_correlated.py:271`) sets `_long_leg` to a plain string market key and stores token IDs under `_token_ids_a/_token_ids_b`. If `_build_legs` is coded for the dict format, it raises `TypeError` on real scanner output. The contradiction guarantees at least one broken production code path.
**Fix:** Pick one canonical format; update scanner, executor, and test fixture to match.

Named `test_replay_returns_existing_id_without_double_execution`; the embedded comment states "the live withdraw is called twice." No assertion checks `gemini.withdraw_usdc.call_count == 1`. A real double-API-call bug — irreversible on-chain transfers — is completely invisible to the test suite.

#### H-11 - Back-all fee base uses per-bet winnings instead of net market profit
**File:** `tests/test_betfair_scan.py:17` (root cause: `fees.py:492-494`) | **Dimension:** bugs

For `probs=[0.30,0.30,0.30]`: `total_cost=0.90`, `gross_spread=0.10`. Correct fee = `5%×0.10 = 0.005`. Implementation computes `net_winnings = 1.0 - min(probs) = 0.70`, giving fee `0.035` (7× too high). Back-all opportunities are systematically over-charged, causing the scanner to reject genuinely profitable arbs.
**Fix:** In `fees.py` change `net_winnings = 1.0 - cheapest` to `net_winnings = gross_spread` (`= 1.0 - total_cost`). Update test assertions accordingly.

#### H-12 - test_backall_uses_cheapest_for_fee asserts wrong fee value
**File:** `tests/test_betfair_scan.py:56` | **Dimension:** bugs

For `probs=[0.10,0.30,0.40]`: `total_cost=0.80`, `gross_spread=0.20`. Correct fee = `5%×0.20 = 0.010`. Test asserts `0.045` (4.5× too high), cementing the broken formula.

#### H-13 - Gemini fee verification uses obsolete min(P,1-P)*rate formula instead of current roundup formula
**File:** `tests/integration/verify_fees.py:245` | **Dimension:** bugs

CLAUDE.md documents the Gemini fee formula changed 2026-03-09 to `roundup(rate × C × P × (1−P))`. The verifier still uses `min(yes_p, 1.0 - yes_p) * GEMINI_FEE_RATE`. If `fees.py` has been updated, every Gemini case reports false FAIL; if not, every case reports false PASS — either way verification is checking the wrong thing.
**Fix:** Implement `expected_fee = math.ceil(GEMINI_TAKER_RATE * yes_p * (1.0-yes_p)) + math.ceil(GEMINI_TAKER_RATE * no_p * (1.0-no_p))` using `GEMINI_TAKER_RATE` from config.

#### H-14 - SSRF via unvalidated SUPABASE_URL env-var in build_client_from_env
**File:** `supabase_sync.py:76` | **Dimension:** security

Every other env-var-controlled URL in the project is validated through `url_guard.assert_public_url()`. `build_client_from_env` skips that step. An attacker who can set Railway env vars could redirect `SUPABASE_SERVICE_KEY`-bearing requests to `http://169.254.169.254/` or any internal host, exfiltrating the high-privilege service key.
**Fix:** `url = assert_public_url(url, env_name='SUPABASE_URL', allow_http=False)` immediately after reading the env var.

---

#### H-15 - Deadlock: should_route_externally acquires self._lock then calls get_best_region which also acquires self._lock
**File:** `latency_monitor.py:372` | **Dimension:** bugs

`RegionRouter` uses `threading.Lock` (non-reentrant). `should_route_externally` acquires `self._lock` at line 366, then calls `self.get_best_region()` at line 372, which tries to acquire the same `self._lock` at line 344. Any call to `should_route_externally` when `GEOGRAPHIC_LATENCY_ENABLED=True` deadlocks the calling thread permanently.
**Fix:** Use `threading.RLock`, or inline the `get_best_region` logic inside `should_route_externally` without re-acquiring the lock.

#### Execution and Scan Bugs

**M-1** `continuous.py:1543` — `FINNHUB_API_KEY` referenced but never imported in continuous mode; every news-snipe scan raises `NameError` swallowed silently. Scan is disabled even when configured.
Fix: Add `FINNHUB_API_KEY` to the inner `from config import ...` statement.

**M-2** `db.py:262` — `get_daily_pnl` adds `SUM(expected_pnl)` from open positions (stale scan-time estimates) to realized P&L. The risk manager's daily loss limit uses this; a losing portfolio with optimistic open estimates can mask real breaches.
Fix: Expose a `realized_only` parameter; the risk manager should use only settled P&L.

**M-3** `kalshi_api.py:165` — HTTP 429 raises `_RateLimitError` which escapes all except clauses, so `_circuit.record_failure()` is never called. After sustained rate limiting the circuit breaker shows zero failures and the scanner continues hammering Kalshi.
Fix: Add `_RateLimitError` to the except clause calling `_circuit.record_failure()`.

**M-4** `event_monitor.py:271` — Platform price is added to the signal aggregator before computing the consensus it is compared against. A two-source consensus halves the apparent divergence; real opportunities are missed or undersized.
Fix: Remove the `add_signal(market_key, platform_name, platform_price)` call; compute external consensus from Metaculus/Manifold only.

**M-5** `calibration_tracker.py:104` — Cache invalidation loop runs after `self._lock` is released, enabling `RuntimeError: dictionary changed size during iteration` under concurrency. `get_weight_multiplier` also has a stale-cache-re-insertion window between the lock release and write-back.
Fix: Add a dedicated `self._cache_lock` and wrap all `_in_memory_cache` operations with it.

**M-6** `polymarket_api.py:73` — Broad `except Exception` catches `_RateLimitError` (HTTP 429) and calls `_circuit.record_failure()`. Three rate-limited requests open the circuit for 30 s, blocking all Polymarket calls when the API is only requesting a slowdown.
Fix: Check `isinstance(e, _RateLimitError)` inside `except Exception` and skip `record_failure()` for that case.

**M-7** `correlation_tracker.py:108` — `build_bucketed_series` outer condition catches `price <= 0 or price >= 1` but the inner `continue` only fires for `price == 0 or price == 1`. Negative prices and prices above 1.0 fall through and corrupt the Pearson correlation series.
Fix: `if price <= 0 or price >= 1: continue` (remove the inner `if` entirely).

**M-8** `scans/kalshi.py:56` — TOCTOU on the Kalshi data cache: the stale-check holds the lock, releases it, then N concurrent threads all fetch simultaneously. `_kalshi_data_cache["ts"] = now` is also captured before the fetch, shortening the effective TTL.
Fix: Hold the lock for the full check-and-fill, or use an in-flight `_fetching` sentinel. Capture `ts` after the fetch completes.

**M-9** `rate_limiter.py:63` — `PlatformCircuitBreaker.is_open()` fully resets on the first post-timeout call (zeros `_failures`, clears `_open_since`), letting all concurrent callers through simultaneously. The documented half-open probe semantics are unimplemented.
Fix: Add a `_half_open` flag; let only the first caller through as a probe; reopen immediately on probe failure.

**M-10** `scans/resolution.py:173` — When `outcomePrices` key exists as `[]`, `.get("outcomePrices", [None, None])[0]` raises `IndexError` (the default is not returned since the key is present).
Fix: `op = market.get("outcomePrices") or [None, None]` then index with length guards.

**M-11** `scans/sxbet.py:55` — `bids[0]["price"]` and `asks[0]["price"]` use direct bracket access; every other exchange scanner uses `.get("price", 0)`. A missing key crashes the scan for that market.
Fix: Use `.get("price")` with a None guard and `continue` on absence.

**M-12** `scans/smarkets.py:86` — `min_depth = min(min_depth, size) if min_depth > 0 else size` resets to a non-zero value after encountering a zero-size runner. Sequence [5, 0, 3] yields 3 instead of 0, inflating reported depth.
Fix: Collect all sizes in a list and call `min(sizes)`.

**M-13** `scans/matchbook.py:91` — Identical broken `min_depth` pattern as M-12.
Fix: Same list-then-`min` approach.

**M-14** `scans/multi_cross.py:447` — `clob_ok` is initialised `True` and never set `False`; the guard `if not clob_ok: continue` is dead code. CLOB failures silently fall through to mid-price profits.
Fix: Set `clob_ok = False` when a price refresh fails, or remove the dead variable and document the fallback.

**M-15** `scans/time_decay.py:81` — `guaranteed_gain = buy_below_price - target_price` yields 0 when both equal `consensus_prob`, and is negative when `consensus_prob > buy_below_price`. The actual convergence gain is `target_price - current_price`.
Fix: `guaranteed_gain = target_price - current_price`.

**M-16** `scans/imbalance.py:82` — Stage 2 collapse check tests magnitude only. A direction reversal (`original_ratio=0.8`, `current_ratio=-0.75`) passes because `abs(-0.75) > 0.56`. A YES signal is sent to the executor when the book has flipped to predict NO.
Fix: `if original_ratio * current_ratio < 0: continue`.

**M-17** `scans/expert_divergence.py:63` — `min_forecasters` is resolved from config but never referenced again; low-quality single-source forecasts pass through at the same confidence as multi-source consensus.
Fix: After `if expert_forecast is None: continue`, add `if expert_forecast.get('num_sources', 0) < min_forecasters: continue`.

**M-18** `scripts/run_edgar_scan.py:196` — `save_seen` trims state by `sorted(seen)[-_MAX_STATE:]`, which sorts by CIK prefix, not recency. Old accessions from low-CIK filers are evicted and will be re-alerted on the next run.
Fix: Sort by `a[11:]` (date-bearing suffix) or use `collections.deque(maxlen=_MAX_STATE)`.

**M-19** `credential_health.py:75` — `check_all_platforms` creates coroutine objects and awaits them serially. 8 platforms x 20 s timeout = 160 s potential latency instead of ~20 s in parallel.
Fix: `tasks = [asyncio.create_task(self._check_platform_health(p)) for p in self.clients]; results_list = await asyncio.gather(*tasks, return_exceptions=True)`.

**M-20** `polygonscan_api.py:119` — `requests.ConnectionError` is caught by `except Exception: return []` before the tenacity `@retry` decorator can retry it. Connection failures are silently discarded.
Fix: Add `except requests.ConnectionError: raise` before the broad handler.

**M-21** `supabase_sync.py:49` — `int(row.get('timestamp', 0))` raises `TypeError` when the column value is SQL NULL (key present, value `None`).
Fix: `ts = int(row.get('timestamp') or 0)`.

**M-22** `signal_aggregator.py:154` — `get_divergences` reads signals twice across two separate lock acquisitions. A signal expiring in the gap produces divergences inconsistent with the consensus value.
Fix: Capture the signal snapshot once in a single lock region and reuse it for both consensus and divergence computation.

**M-23** `scripts/analytics.py:106` — Sharpe SQL: `stddev * SQRT(252)` is annualized volatility, not Sharpe ratio. The `avg_pnl` numerator is absent; high-volatility strategies show large "Sharpe" even with negative returns.
Fix: `(avg_pnl / NULLIF(stddev, 0)) * SQRT(252) as annual_sharpe`.

**M-24** `scripts/analytics.py:109` — Max drawdown computed as `max_cumulative_pnl - min_cumulative_pnl` (total range). Equals total profit for a trending strategy; not peak-to-subsequent-trough.
Fix: Window function: `MAX(running_max_pnl - cumulative_pnl)` where `running_max_pnl = MAX(cumulative_pnl) OVER (... ORDER BY timestamp ROWS UNBOUNDED PRECEDING)`.

**M-25** `scripts/analytics.py:199` — `print("\n".join(headers))` prints each column header on its own line. Data rows use a single fixed-width format; the table output is completely misaligned.
Fix: Print header as one line matching the data row format.

**M-26** `superforecaster_api.py:59` — `_get_cached` and `_set_cached` have no lock. Two concurrent threads can both find the entry expired and both call `del self._cache[key]`; the second raises `KeyError`.
Fix: Add `threading.Lock`, or use `self._cache.pop(key, None)` (GIL-atomic).

**M-27** `scans/cross_category.py:99` — `ExternalSignalFetcher._get_cached` has the same unguarded check-then-delete race as M-26.
Fix: `entry = self._cache.pop(key, None); if entry and time.time() <= entry.get("expires", 0): return entry.get("value")`.

**M-28** `scans/cross_category.py:189` — `_calculate_implied_prob` declares and documents `days_remaining` but never uses it. A 1-day and 365-day market at the same price return identical probabilities.
Fix: Scale distance thresholds by `max(1, sqrt(days_remaining / 30))` or similar time-decay factor.

**M-29** `scripts/analytics.py:68` — DuckDB connection opened at line 68; `conn.close()` at line 115 is only reached on the success path. Any exception leaks the open connection and its file lock.
Fix: Wrap in `try/finally` or `with duckdb.connect(...) as conn:`.

**M-30** `market_discovery.py:275` — External market titles from Polymarket and Kalshi APIs are embedded verbatim as the LLM user message. A market titled `[click](https://attacker.com)` can inject Telegram-rendered links or adversarial instructions into the judgement call.
Fix: Add an untrusted-content boundary to the system prompt; strip angle-brackets and newlines; truncate to 250 chars.

**M-31** `ws_feeds.py:866` — `BETFAIR_STREAM_HOST` env var is passed directly to `asyncio.open_connection` without `assert_public_url`. An attacker-controlled hostname with a matching TLS cert would receive the Betfair `app_key` and `session_token` in the authentication handshake.
Fix: `assert_public_url(f"https://{BETFAIR_STREAM_HOST}", env_name="BETFAIR_STREAM_HOST", allow_http=False)` before connecting.

#### Infrastructure / Security — Medium

**M-32** `dashboard.py:294` — Neither `_send_html` nor `_send_json` emits `X-Frame-Options`, `Content-Security-Policy: frame-ancestors`, or `X-Content-Type-Options`. Kill-switch and rebalance POST endpoints are exposed; without `X-Frame-Options: DENY` a malicious page can iframe the dashboard for clickjacking.
Fix: Add `X-Frame-Options: DENY`, `Content-Security-Policy: frame-ancestors 'none'`, and `X-Content-Type-Options: nosniff` to both send helpers.

**M-33** `gemini_api.py:565` — `withdraw_usdc` accepts any non-empty string as the destination address. On-chain USDC withdrawals are irreversible. Neither `withdraw_usdc` nor its callers in `treasury.py` validate EVM address format.
Fix: `if not re.match(r'^0x[0-9a-fA-F]{40}$', address): return None` before building the payload.

**M-34** `sxbet_api.py:16` — `SXBET_API_BASE_URL` is the only env-var-controlled API base URL not covered by `assert_public_url`. All other clients apply this guard.
Fix: `SXBET_API_URL = assert_public_url(os.getenv("SXBET_API_BASE_URL", "https://api.sx.bet"), env_name="SXBET_API_BASE_URL", allow_http=False)`.

**M-35** `sentry_init.py:18` — `traces_sample_rate=1.0` for all non-production environments exfiltrates opportunity dicts (cross-platform prices, market tickers, `net_profit`) to Sentry on every transaction.
Fix: Cap at `0.05` unconditionally, or register a `before_send` callback stripping `net_profit`, `_token_ids`, `prices` keys.

**M-36** `scans/whale_copy.py:66` — `int(tx.get("timeStamp", 0))` raises `ValueError` on non-numeric API values. The exception is caught at the wallet level and `last_block_cache` is never updated, permanently abandoning the wallet on every subsequent scan cycle.
Fix: Wrap per-transaction parsing in its own `try/except ValueError`.

**M-37** `scans/cross_category.py:126` — CoinGecko BTC and ETH prices are parsed and fed directly into trading decisions without type, range, or sanity validation.
Fix: `if not isinstance(price, (int, float)) or not (MIN_BTC <= price <= MAX_BTC): raise ValueError(...)`.

**M-38** `tests/test_url_guard.py:116` — `ALLOW_PRIVATE_INTERNAL_URLS=true` disables all SSRF protection simultaneously, including cloud-metadata addresses. One accidentally-copied dev env var eliminates the entire guard for all platform clients.
Fix: Replace with a per-variable allowlist: `ALLOW_PRIVATE_URLS_FOR=POLYGON_RPC_URL`.

**M-39** `tests/test_risk_manager.py:342` — `test_zero_balance_returns_zero` asserts `result == 5.0`, contradicting its own name. Zero-balance callers can receive positive trade sizes; a fix to the implementation would fail this test for the wrong reason.

**M-40** `tests/test_credential_health.py:179` — `test_timeout_is_info_severity` wraps assertions in `if self.mock_alert_manager.alert.called:`. Zero alerts fired → block skipped → test passes vacuously.

**M-41** `tests/test_credential_health.py:207` — Same conditional-guard anti-pattern as M-40 for the auth-failure WARNING alert.

**M-42** `tests/test_analytics.py:167` — `max_drawdown` expected value `0.15` is only reachable via the broken `max - min` formula. Correct peak-to-trough value for the test sequence is `0.10`.

A market titled `[click](https://attacker.com)` on any monitored platform would render as a clickable link in the operator's Telegram bot messages, creating a phishing vector against the bot owner. `opp['type']` is safely backtick-quoted but `market` and `roi` are not.
