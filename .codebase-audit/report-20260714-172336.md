## Critical Findings

### C-1 · TaxOptimizer.get_harvest_candidates deadlocks on Lock re-entry
**File:** `capital_optimizer.py` **Lines:** 302–314

`self._lock = threading.Lock()` (line 208 — non-reentrant). `get_harvest_candidates` acquires it at line 302, then calls `self.is_long_term()` at line 314, which also calls `with self._lock:` at line 256. Python's non-reentrant lock blocks the same thread forever. Fires whenever `TAX_AWARE_ENABLED=True` and any position passes the loss threshold — exactly when harvest logic is most needed.

**Fix:** `self._lock = threading.RLock()` in `TaxOptimizer.__init__`, or extract a `_is_long_term_unlocked()` helper that reads `self._cost_basis` directly without re-acquiring the lock.

---

### C-2 · RegionRouter.should_route_externally deadlocks on Lock re-entry
**File:** `latency_monitor.py` **Lines:** 366–372

`self._lock = threading.Lock()` (line 312). `should_route_externally` acquires at line 366, then calls `self.get_best_region()` at line 372, which acquires the same lock at line 344. Thread blocks forever. Fires whenever `GEOGRAPHIC_LATENCY_ENABLED=True`.

**Fix:** `self._lock = threading.RLock()` in `RegionRouter.__init__`, or extract a `_get_best_region_locked()` private helper that assumes the lock is already held.

## High Findings

### Security — Fund Safety

#### H-1 · `~/.claude/.env` loaded into trading app environment
**File:** `config.py:9`, `cli.py:107`

`load_dotenv(os.path.expanduser("~/.claude/.env"))` in both files silently promotes the Claude coding-agent's personal secrets into `os.environ`, potentially supplying or corrupting platform credentials used for live order placement.

**Fix:** Remove both calls. Use `load_dotenv()` (project root `.env`) for local dev fallback.

---

#### H-2 · Ethereum deposit addresses accepted without format validation
**File:** `treasury.py:214`

Both deposit addresses are validated only for non-emptiness. `GEMINI_DEPOSIT_ADDRESS` is not registered in `config.py` and is never evaluated by `validate_config()` at startup. A typo or Railway config injection directs an irreversible on-chain USDC transfer to the wrong address.

**Fix:** Add `re.compile(r'^0x[0-9a-fA-F]{40}$')` validation in `TreasuryManager.__init__` for both addresses. Register `GEMINI_DEPOSIT_ADDRESS` in `config.py`.

---

#### H-3 · `withdraw_usdc` sends funds without validating destination address format
**File:** `gemini_api.py:576`

Only `if not address:` is checked. Any non-empty string — whitespace-padded, wrong-network, truncated — passes before initiating the irreversible on-chain transfer.

**Fix:** `return bool(resp and resp.get("isCancelled"))` for the cancel logic. For withdraw: assert `r'^0x[0-9a-fA-F]{40}$'` before building the payload.

---

#### H-4 · Idempotency key does not prevent duplicate live fund transfers
**File:** `treasury.py:172`

`db.log_transfer` silently returns the existing row id on duplicate key collision, but `execute_transfer` has no replay guard between `log_transfer` and `_gemini_to_polymarket`. A network retry or double POST to `/api/rebalance/execute` transfers funds twice. `test_replay_returns_existing_id_without_double_execution` (H-21) openly documents this and lacks the `assert_called_once()` assertion.

**Fix:** After `log_transfer`, query the existing row's status. If already `'succeeded'`, return early without calling the corridor method.

### Fee Calculation Bugs

#### H-5 · `_select_fees()` probability weights are swapped
**File:** `fees.py:34`

```python
return prob_a * case_b_fees + (1.0 - prob_a) * case_a_fees  # operands reversed
```

`prob_a` weights `case_b_fees` and vice versa. Affects `net_profit_cross_betfair`, `net_profit_cross_smarkets`, `net_profit_cross_generic`, and `net_profit_triangular` under `FEE_MODEL='expected_value'`.

**Fix:** `return prob_a * case_a_fees + (1.0 - prob_a) * case_b_fees`

---

#### H-6 · `net_profit_betfair_backall()` overstates commission by ~7×
**File:** `fees.py:492`

```python
net_winnings = 1.0 - min(implied_probs)   # wrong: payout on cheapest runner only
fee = betfair_commission(net_winnings, commission_rate)
```

Commission base should be `gross_spread = 1 - sum(implied_probs)` (already computed on line 486). Using `1 - min_price` overstates it by ~7× on a 3-runner market, rejecting all valid back-all arbs.

**Fix:** `fee = betfair_commission(gross_spread, commission_rate)`

#### H-7 · `net_profit_smarkets_backall()` has the same commission overstatement
**File:** `fees.py:552`

Identical logic flaw for the Smarkets 2% commission path. **Fix:** Same as H-6.

### Data / Calculation Bugs

#### H-8 · `_extract_net_roi` returns 100× correct value for ROI strings ≤ "1.00%"
**File:** `position_sizer.py:355`

`abs(1.0) > 1` is `False`, so `"1.00%"` → 1.0 (100%) instead of 0.01 (1%). The Kelly formula allocates `max_fraction` of bankroll to near-zero-edge opportunities.

**Fix:** Track the `%` suffix before stripping and always divide by 100 when it was present.

---

#### H-9 · Platform price included in signal-aggregator consensus, suppressing divergence signals
**File:** `event_monitor.py:271`

`add_signal(market_key, platform_name, platform_price)` is called before `get_consensus()`. The platform's own price pulls the consensus toward itself; apparent divergences are ~43% smaller than reality for a typical 30%/60% split.

**Fix:** Remove the platform's `add_signal` call. Only external sources (Metaculus, Manifold) should populate the aggregator used for divergence comparison.

---

#### H-10 · `_refine_cross_all_with_clob` is a no-op — CLOB filter never removes candidates
**File:** `scans/cross.py:621`

`refined_out = []` is declared but never appended to and never returned. The function has no return statement. The call site ignores the return value. Every other `_refine_*_with_clob` function builds and returns a filtered list; this one uniquely fails to, making the stage-2 CLOB filter completely inert for cross-all Polymarket opportunities.

**Fix:** Add `refined_out.append(o)` inside the passing `if` block; update the caller to use the filtered result.

---

#### H-11 · Triangular CLOB refinement uses non-existent `_price_b`/`_price_a` keys (always zero)
**File:** `scans/triangular.py:414`

`o.get("_price_b", 0)` — these keys are never set in `scan_triangular`'s opp dict. `other_price` is always 0, producing a phantom gross spread of ~0.4–0.6. Every Polymarket-containing triangular candidate survives refinement with inflated profit.

**Fix:** Add `"_price_a": best_yes, "_price_b": best_no` to the opp dict in `scan_triangular`.

---

#### H-12 · Wrong CLOB side used when Polymarket is the NO leg of a triangular arb
**File:** `scans/triangular.py:422`

`_side_b` is never set → `pm_side` defaults to `"yes"` → `clob["yes_ask"]` is used for the NO leg. Wrong price, wrong profit, wrong execution.

**Fix:** Add `"_side_a": "yes", "_side_b": "no"` to the opp dict in `scan_triangular`.

---

#### H-13 · CLOB fetch tasks built with swapped (key, market) pairs in correlated refinement
**File:** `scans/correlated.py:368`

When `market_b` is the cheaper (long) leg, `_long_market=market_b` but `_market_key_a` still refers to `market_a`. The fetch pairs the wrong key with the wrong market; CLOB books for long and short legs are swapped. Affects 50% of correlated pairs.

**Fix:** Use `(opp.get("_long_leg"), opp.get("_long_market"))` and `(opp.get("_short_leg"), opp.get("_short_market"))` in the fetch loop.

---

#### H-14 · `outcomePrices` items used without float conversion in rewards scan fallback
**File:** `scans/rewards.py:231`

Polymarket's API returns `outcomePrices` as JSON strings. In the fallback path, `mid_price = prices[0]` (a string) causes `TypeError: unsupported operand type(s) for -: 'str' and 'float'` in `_calculate_optimal_quotes`.

**Fix:** `mid_price = float(prices[0])` with a `try/except (TypeError, ValueError): continue` guard.

---

#### H-15 · First-match-wins in `_match_rule` selects wrong correlation rule when keywords overlap
**File:** `scans/cross_category.py:243`

"Will Bitcoin (BTC) reach $150,000?" scores 2 matches against `btc_100k` (first rule) and is returned immediately, despite `btc_150k` scoring 3 matches. Implied probability is calculated against the wrong threshold.

**Fix:** Return the rule with the highest keyword match count, not the first rule exceeding the minimum threshold.

---

#### H-16 · `CrossPairIndex.evaluate` returns `_kalshi_yes`/`_kalshi_no` semantically swapped for inverted pairs
**File:** `cross_pair_index.py:280`

After the local `k_yes, k_no = k_no, k_yes` swap for profit calculation, the returned dict assigns the post-swap (wrong) values to `_kalshi_yes` and `_kalshi_no`. The executor may place orders on the wrong Kalshi side.

**Fix:** Save pre-swap originals; use them for the returned dict fields.

---

#### H-17 · `gemini_api.py cancel_order` returns `True` for any non-`None` response
**File:** `gemini_api.py:547`

`return resp is not None` fires for error responses where `isCancelled` is absent or `False`, reporting cancellation success when the order was not cancelled. Leads to double-execution or failure to clean up filled orders.

**Fix:** `return bool(resp and resp.get("isCancelled"))`

---

#### H-18 · Race condition on daily-loss alert flags — CRITICAL alert can be silently dropped
**File:** `alerting.py:192`

`_loss_80_fired`/`_loss_100_fired` are read and written outside `self._lock`. Two executor threads can both evaluate `not self._loss_100_fired` as `True` before either sets it; the WARNING can acquire the shared rate-limit slot first, causing the CRITICAL to be suppressed and never retried.

**Fix:** Protect the entire flag-check-and-set inside `self._lock`. Or use separate rate-limit keys for WARNING and CRITICAL.

---

#### H-19 · `RedditClient._authenticate` sets `Authorization: Bearer None` on partial OAuth response
**File:** `reddit_api.py:111`

If Reddit returns HTTP 200 but a body missing `access_token`, `data.get("access_token")` returns `None`, `self.authenticated = True` is set, and `Authorization: Bearer None` is written to session headers. Every subsequent API call returns 401; subsequent `_authenticate()` calls re-enter the token-fetch branch on every call.

**Fix:** `if not self._access_token: logger.warning(...); return False`

---

#### H-20 · Sharpe ratio formula computes annualized volatility, not Sharpe ratio
**File:** `scripts/analytics.py:106`

```sql
THEN stddev * SQRT(252)   -- should be (avg_pnl / NULLIF(stddev, 0)) * SQRT(252)
```

`avg_pnl` is already computed in the same CTE but never used in the Sharpe expression. A strategy with large variance and negative average returns displays a large positive "Sharpe".

**Fix:** `THEN (avg_pnl / NULLIF(stddev, 0)) * SQRT(252)`

---

#### H-21 · `test_replay_returns_existing_id_without_double_execution` never asserts `call_count`
**File:** `tests/test_treasury.py:176`

The embedded comment says "the live withdraw is called twice" but there is no `assert_called_once()` assertion. CI passes green while 200 USD would be withdrawn instead of 100. Exposes the production bug in H-4.

**Fix:** Add `gemini.withdraw_usdc.assert_called_once_with(address="0xPMproxy123", amount=100.0)`.
