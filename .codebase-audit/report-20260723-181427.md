## High Severity Findings

### H-01 â€” BID prices returned as ASK prices on SX Bet, generating false arb signals
**File:** `sxbet_api.py:217` | **Dimension:** bugs

`get_market_price()` selects the highest YES BID and highest NO BID and returns them as market prices. In SX Bet's P2P model, `isMakerBettingOutcomeOne=True` means the maker is bidding on YES; the ASK price a taker pays is `1 - best_NO_bid`, not the best YES bid. A market with YES_bid=0.40 and NO_bid=0.40 reports a combined price of 0.80 (signalling a 20% arb gap) when true ask prices sum to 1.20 (overround). Every cross-platform scan involving SX Bet produces structurally wrong profit estimates.

**Fix:** After collecting bids: `yes_ask = 1.0 - best_no_bid; no_ask = 1.0 - best_yes_bid`. Return derived ask prices.

### H-02 â€” position_sizer.py: `_extract_net_roi` returns 100x correct value for sub-1% ROI strings
**File:** `position_sizer.py:355` | **Dimension:** bugs

All scan modules write `net_roi` as a percentage string (e.g., `"0.50%"`). After stripping `%`, `"0.50"` parses to float `0.5`. Because `abs(0.5) > 1` is `False`, the function returns `0.5` â€” treating 0.5% ROI as 50%. Kelly position sizing is computed against this 100x inflated edge for virtually every real opportunity (which has ROI < 1%), producing massively oversized orders.

**Fix:** `if raw_str.endswith('%'): return float(raw_str[:-1]) / 100.0`. Remove the broken `abs(val) > 1` heuristic.

### H-03 â€” `credential_health.py`: `check_all_platforms()` awaits coroutines sequentially instead of concurrently
**File:** `credential_health.py:74` | **Dimension:** bugs

Builds a list of unawaited coroutines then awaits them one-by-one. With 8 platform clients, 2 retries, and 10-second timeouts, a full health check blocks up to 160 seconds, staling timestamps and triggering false CRITICAL alerts.

**Fix:** `results = await asyncio.gather(*[self._check_platform_health(p) for p in self.clients], return_exceptions=True)`.

### H-04 â€” Reddit `_authenticate` sets "Bearer None" header when `access_token` absent from response
**File:** `reddit_api.py:111` | **Dimension:** bugs

If the API returns HTTP 200 without `access_token`, `data.get("access_token")` returns Python `None`. The f-string produces literal `"Bearer None"`. `self.authenticated = True` is set, every subsequent call carries the invalid header, Reddit returns 401/403, all methods return `[]`. The re-entry guard never fires, hammering the auth endpoint indefinitely.

**Fix:** After `self._access_token = data.get("access_token")`, add: `if not self._access_token: logger.warning(...); return False`.

### H-05 â€” `flip_lane` and `rotate_secret` intents can be double-executed after `append_event` persistence failure
**File:** `broker/broker.py:244` | **Dimension:** bugs

When `append_event(STATUS_EXECUTED)` fails, the except block escalates and freezes capital for `move_capital` only. For `flip_lane` and `rotate_secret`, execution continues and returns `STATUS_EXECUTED` while the DB row shows `STATUS_PENDING`. A subsequent `process_stored()` on restart re-executes the intent â€” double-rotating secrets or double-flipping lanes.

**Fix:** After escalating for non-capital intents, record `STATUS_IN_DOUBT` event (best-effort) and return `BrokerDecision(intent_id, STATUS_IN_DOUBT, str(exc))`.

### H-06 â€” `process_stored` crashes with `AttributeError` if `get_intent` returns `None`
**File:** `broker/broker.py:120` | **Dimension:** bugs

TOCTOU window between `current_status(intent_id)` and `get_intent(intent_id)`. If the intent is deleted between calls, `get_intent` returns `None`. `_process_created(intent_id, None)` raises `AttributeError: 'NoneType' has no attribute 'payload'`, crashing the BrokerWorker thread.

### H-07 â€” `_refine_cross_with_clob` always stores `min(pm_yes_depth, pm_no_depth)` regardless of which leg was selected
**File:** `scans/cross.py:149` | **Dimension:** bugs

After selecting the winning strategy (`result1` or `result2`), the code always stores `min(pm_yes_depth or 0, pm_no_depth or 0)` as `_clob_depth`. If `pm_yes_depth=1000`, `pm_no_depth=0`, and `result1` wins, the correct depth is 1000 but `_clob_depth` is set to 0, causing the executor to skip or zero-size the trade.

**Fix:** `opp["_clob_depth"] = (pm_yes_depth or 0) if best == result1 else (pm_no_depth or 0)`.

### H-10 â€” `annual_sharpe` returns annualised volatility, not Sharpe ratio
**File:** `scripts/analytics.py:106` | **Dimension:** bugs

`stddev * SQRT(252)` computes annualised volatility. Sharpe ratio requires `(avg_pnl / stddev) * SQRT(252)`. A losing high-volatility strategy and a winning high-volatility strategy report identical "Sharpe" values. The metric is used for performance evaluation.

**Fix:** `THEN (avg_pnl / NULLIF(stddev, 0)) * SQRT(252)`.

### H-15 â€” SSRF via unvalidated `SUPABASE_URL` env var in `SupabaseControlsClient`
**File:** `mm_pilot.py:241` | **Dimension:** security

### H-17 â€” Bracket arb sum omits unpriced brackets, producing false underround signals
**File:** `scans/bracket.py:176` | **Dimension:** bugs

## Medium Severity Findings

### M-01 â€” `_seq_counter` race condition causes `PriorityQueue` dict-comparison `TypeError` | `continuous.py:1218`
Plain nonlocal integer read and incremented by concurrent WebSocket threads without a lock. Two threads observing the same seq value cause `asyncio.PriorityQueue` to compare dicts, raising `TypeError`. **Fix:** Protect with `threading.Lock` or `itertools.count()` wrapped in a lock.

### M-02 â€” `_build_legs` raises `ValueError` for signal-strategy types instead of returning `[]` | `executor.py:2079`
For `Imbalance`, `NewsSnipe`, `TimeDecay`, `Correlated`: raises `ValueError` when `_token_ids` is missing. `execute()` has no `try/except` around `_build_legs`. The `ValueError` is swallowed by the priority consumer without a DB record. **Fix:** Wrap in `try/except ValueError: return False` or change branches to `return []`.

### M-03 â€” `--max-trade 0` silently replaced by env-var default (falsy-zero) | `cli.py:1223`
`args.max_trade = 0.0` is falsy; `or` falls through to 5.0. An operator passing `--max-trade 0` to disable execution gets 5.0 instead â€” a silent safety bypass. **Fix:** `max_trade = args.max_trade if args.max_trade is not None else float(os.getenv(...))`. Set argparse default to `None`.

### M-04 â€” `scan_logical_arb` receives a list where it expects a dict | `continuous.py:1984`
`poly_markets` (a `list[dict]`) is passed as `markets_by_key`. If `scan_logical_arb` calls `.items()` or dict-style access, raises `AttributeError`/`TypeError`, swallowed silently, making logical-arb permanently produce zero opportunities. **Fix:** Build the dict pre-call, mirroring the imbalance scan pattern.

### M-05 â€” `requests.Timeout` swallowed in `_private_request` â€” timeout retries never fire | `gemini_api.py:194`
`Timeout` is listed in the tenacity decorator but not in the re-raise clause. Falls through to `except requests.RequestException: return None`. Tenacity never sees the exception. **Fix:** Add `except requests.exceptions.Timeout: _circuit.record_failure(); raise` between `ConnectionError` and broad `RequestException` blocks.

### M-06 â€” `fetch_incentive_programs` uses `next_cursor` key while all other paginating methods use `cursor` | `kalshi_api.py:608`
`data.get("next_cursor")` always returns `None` if the API uses `"cursor"`. Loop exits after the first page; all subsequent incentive program pages are silently dropped. **Fix:** `cursor = data.get("cursor")`.

### M-07 â€” Platform price self-contaminates the consensus it is compared against | `event_monitor.py:271`
Platform's own price is added as a signal source, then `get_consensus()` averages it with external sources. Comment claims "excluding the platform's own price" but code does the opposite. True divergences are systematically underestimated. **Fix:** Remove `self.signal_aggregator.add_signal(market_key, platform_name, platform_price)`, or call `get_consensus()` before adding the platform signal.

### M-08 â€” Nested `if` structure passes prices outside `(0,1)` through boundary filter | `correlation_tracker.py:108`
For price `-0.001`, outer `if price <= 0` is `True` but inner `if price == 0` is `False`, so `continue` never fires. Invalid prices corrupt the Pearson correlation series. **Fix:** `if not (0 < price < 1): continue`.

### M-09 â€” `InventoryTracker.can_trade` TOCTOU race between `get_position` and `get_total_exposure` | `market_maker.py:88`
Both methods acquire `self._lock` independently. A concurrent fill between calls can push the total above the limit undetected, allowing two threads to both pass the check and together exceed `max_total`. **Fix:** Merge into a single `with self._lock:` block that reads both values atomically.

### M-10 â€” `KalshiRewardTracker.log_order_cancelled` holds lock while calling a blocking DB operation | `market_maker.py:1056`
The `with self._lock:` region spans from the guard check through `self._db.log_reward_metric(...)`. Under high cancellation rates, a slow DB write stalls all other tracker operations. **Fix:** Pop the order inside the lock, exit the `with` block, then call the DB method.

### M-11 â€” `_in_memory_cache` mutated outside lock, racing with lock-free reads | `calibration_tracker.py:104`
Lock is released after DB commit; cache invalidation loop runs outside. Concurrent `get_weight_multiplier` reads `_in_memory_cache` without any lock. The non-atomic `if key in dict: value = dict[key]` pattern raises `KeyError` if a key is deleted between checks. **Fix:** Move cache invalidation inside `with self._lock:` block.

### M-12 â€” `record_snapshot` aborts entire batch on malformed `total_cost` string | `snapshot.py:163`
`float(total_cost_str.replace("$", ""))` has no `try/except`. A single non-numeric value raises `ValueError`, exits the loop, and discards all accumulated rows. **Fix:** Wrap per-opportunity block in `try/except (ValueError, TypeError)` with `continue`.

### M-13 â€” `analyze_sentiment` returns weighted sums as `positive_count`/`negative_count`, not post counts | `reddit_api.py:329`
Documented as post counts but are engagement-weighted sums truncated to `int`. A single high-engagement post (score=5000) contributes weight 70, reported as if 70 positive posts were found. Same bug in `twitter_api.py:250`. **Fix:** Track actual post counts separately from the weighted signal.

`with self._db._lock:` covers only the `SELECT MIN` query. The `else`-branch `SELECT MAX` runs without holding the lock, racing against concurrent writes. **Fix:** Extend the lock context to cover both queries.

### M-15 â€” `supabase_queue._assert_stored_matches` JSON string comparison fragile across JSONB round-trip | `broker/supabase_queue.py:188`
PostgreSQL JSONB normalises `float 1.0` to `int 1`. `json.dumps(1.0)` is `"1.0"` but `json.dumps(1)` is `"1"`.
